# ADR-002: 降低集群配置的优先级

| 项 | 值 |
|----|----|
| **日期** | 2026-04-14 |
| **作者** | smart-mqtt 产品团队 |
| **状态** | accepted |
| **关联版本** | v1.6.0+ |

---

## 1. 背景与问题

原 v1.6.0 规划中，"集群配置向导"被列为 Should Have 优先级。

但根据真实用户分层分析：
- 90% 的用户：单实例足够支撑业务
- 只有不到 10% 的用户：真正具备集群体量的业务规模
- 喊"要集群"的用户，大多数其实还没到那个体量

## 2. 驱动因素

### 用户洞察修正
1. INS-001 的重新解读：用户卡在集群配置是因为"试一下"，而不是生产环境需要
2. 真正需要集群的 10% 企业用户，具备自行配置的技术能力
3. 新手用户的核心痛点：**不是集群难配，而是不知道自己不需要集群**

### 资源分配视角
1. 集群向导开发成本高，但受益面窄（<10%）
2. 同样的投入可以让 90% 的单实例用户体验提升更多
3. 避免"为了 10% 用户的玩具需求，增加 90% 用户的认知负担"

## 3. 备选方案评估

### 方案 A: 维持原优先级，做集群配置向导
- ✅ 优点：解决 INS-001 的表层痛点
- ❌ 缺点：资源错配，90% 用户用不上，给新手造成"必须搭集群"的错误认知
- 💸 ROI: 低

### 方案 B: 降低优先级 + 前置"你需要集群吗？"判断
- ✅ 优点：90% 的新手不再被误导，资源投到更广泛的用户价值
- ✅ 优点：文档首页明确说明"什么时候需要集群"
- ❌ 缺点：少数真正需要集群的用户还是要走老路
- 💸 ROI: 高

### 方案 C: 完全砍掉集群功能
- ✅ 优点：产品更简洁
- ❌ 缺点：失去高端用户的可能性
- 💸 ROI: 不可取

## 4. 最终决策

**选择方案 B: 降低集群配置优先级 + 增加场景引导**

### 决策依据
1. **ROI 最大化**：同样投入服务 90% 用户 vs 10% 用户
2. **认知减负**：新手不要一上来就面对"集群"这个概念
3. **用户分层**：真正需要集群的用户有能力自己搞定
4. **产品聚焦**：先把单实例体验做到极致

## 5. 后果与影响

### 正面影响
- v1.6.0 资源更聚焦，交付质量更高
- 新手用户不再被"集群"概念吓到
- 避免了过早优化的陷阱

### 负面影响/应对
- 少数真正需要集群的用户：文档增加专门的集群部署指南即可
- INS-001 部分未解决：这是有意识的选择，接受这个 trade-off

## 6. 后续跟进

- [x] 用户洞察 INS-001 修正解读
- [x] v1.6.0 集群配置向导降为 Could Have
