# ADR-001: smart-mqtt 不内置消息持久化

| 项 | 值 |
|----|----|
| **日期** | 2026-04-14 |
| **作者** | smart-mqtt 产品团队 |
| **状态** | accepted |
| **关联版本** | v1.6.0+ |

## 1. 背景与问题

MQTT 协议规范中定义了 QoS 1/2 消息的持久化要求。行业内绝大多数 MQTT Broker（如 EMQX、Mosquitto 等）都内置了消息持久化功能。

但是 smart-mqtt 面临一个核心定位选择：
- 做一个大而全的 Broker
- 还是聚焦于"高性能消息路由"这个核心价值？

## 2. 驱动因素

### 业务驱动
1. smart-mqtt 的核心优势：**极致性能**和**轻量级**
2. 绝大多数物联网场景：消息都是实时转发，不需要回溯
3. 持久化引入的复杂度会拖慢核心路由性能

### 技术驱动
1. 持久化是 Broker 性能的头号杀手
2. 不同用户的持久化需求差异极大（有的要 Kafka、有的要 Redis、有的要数据库）
3. 内置持久化永远无法满足所有用户的个性化需求

## 3. 备选方案评估

### 方案 A: 内置消息持久化
- ✅ 优点：开箱即用，符合用户预期
- ❌ 缺点：性能下降 50%+，代码复杂度 ×3，维护成本高
- 💸 成本：需要 3 人月专门开发和优化

### 方案 B: 完全不做持久化，通过插件桥接
- ✅ 优点：极致路由性能，架构简洁，用户按需选择持久化方案
- ✅ 优点：可以桥接 Kafka/RocketMQ/Redis 等各种专业存储
- ❌ 缺点：部分用户预期落空，需要文档明确说明
- 💸 成本：0 开发，只需要提供桥接示例

### 方案 C: 极简内存持久化
- ✅ 优点：满足协议要求，性能影响小
- ❌ 缺点：用户会误认为有磁盘持久化，造成数据损失
- 💸 成本：中等

## 4. 最终决策

**选择方案 B: 完全不内置消息持久化，通过插件桥接**

### 决策依据
1. **定位差异**：这是 smart-mqtt 与其他 Broker 最核心的差异化优势
2. **性能优先**：把所有资源聚焦在"路由性能"这个北极星指标上
3. **架构简洁**：do one thing, do it well
4. **用户自由**：用户可以根据场景选择最适合的持久化方案

## 5. 后果与影响

### 正面影响
- 路由性能达到理论极限
- 代码库保持轻量简洁
- 避免了"持久化技术栈选型"的永久债务
- 形成鲜明的产品定位："极致性能的纯消息路由器"

### 负面影响/应对
- 用户教育成本：**文档首页必须醒目说明这个定位**
- 离线消息场景：提供 Redis 桥接插件示例
- 协议合规性：明确标注"QoS 1/2 仅保证传输链路可靠性，不保证服务端持久化"
