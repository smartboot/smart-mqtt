`bench-plugin` 是一个 MQTT Broker 性能压测插件，支持 Publish（发布压测）和 Subscribe（订阅压测）两种场景。

## 功能特性

- **双场景压测**: 支持发布压测和订阅压测两种模式
- **高并发支持**: 可配置数千个并发连接
- **实时监控**: 每5秒输出 TPS 统计数据
- **灵活配置**: 支持 QoS、主题数、消息大小等参数配置
- **动态启停**: 支持通过管理页面动态启动和停止压测

## 快速开始

### 1. 配置压测参数

通过管理页面配置 `plugin.yaml` 文件：

```yaml
# 压测场景: publish (发布压测) | subscribe (订阅压测)
scenario: publish

# ==================== 公共配置 ===================
# MQTT 服务器地址
host: 127.0.0.1
# MQTT 服务器端口
port: 1883
# 消息负载大小（字节）
payloadSize: 1024
# 主题数量
topicCount: 128
# QoS 等级: 0 (AtMostOnce) | 1 (AtLeastOnce) | 2 (ExactlyOnce)
qos: 0

# ==================== 发布压测配置 ===================
publish:
  # 发布者数量（并发连接数）
  connections: 1000
  # 每次发布的消息数量
  publishCount: 1
  # 发布间隔（毫秒），设为 0 可实现最大吞吐量
  period: 1

# ==================== 订阅压测配置 ===================
subscribe:
  # 订阅者数量（并发连接数）
  connections: 1000
  # 发布者数量（用于向主题发送消息，0 表示不启动发布者）
  publisherCount: 1
  # 每次发布的消息数量
  publishCount: 1
  # 发布间隔（毫秒）
  publishPeriod: 1
```

### 2. 启动压测

插件安装后将自动进入就绪状态，可通过管理页面动态启动压测。启动后有 5 秒延迟开始执行压测任务。

### 3. 停止压测

在管理页面点击停止按钮，插件会立即停止压测并断开所有连接。

### 4. 查看结果

压测过程中可以通过以下方式查看结果：

- **终端输出**: 每 5 秒输出一次压测统计：
  ```
  total: 50000	TPS: 10000
  ```
- **后台系统 Dashboard**: 登录管理后台后，在 Dashboard 页面可以查看更详细的实时图表，包括 TPS 趋势图、连接数、消息触达率等指标

## 参数说明

### 公共参数（PluginConfig）

| 参数 | 说明 | 默认值 | 取值范围 |
|------|------|--------|----------|
| `scenario` | 压测场景 | `publish` | `publish` / `subscribe` |
| `host` | MQTT 服务器地址 | `127.0.0.1` | IP 地址或域名 |
| `port` | MQTT 服务器端口 | `1883` | 有效端口号 |
| `payloadSize` | 消息负载大小（字节） | `1024` | 0 - 1048576 |
| `topicCount` | 主题数量 | `128` | 1 - 10000 |
| `qos` | QoS 等级 | `0` | `0` / `1` / `2` |

### 发布压测配置（PublishConfig）

| 参数 | 说明 | 默认值 | 取值范围 |
|------|------|--------|----------|
| `connections` | 发布者数量（并发连接数） | `1000` | 1 - 100000 |
| `publishCount` | 每次发布的消息数量 | `1` | 1 - 1000 |
| `period` | 发布间隔（毫秒） | `1` | 0 - 60000 |

> 💡 **提示**: 将 `period` 设为 `0` 可实现最大吞吐量压测

### 订阅压测配置（SubscribeConfig）

| 参数 | 说明 | 默认值 | 取值范围 |
|------|------|--------|----------|
| `connections` | 订阅者数量（并发连接数） | `1000` | 1 - 100000 |
| `publisherCount` | 发布者数量 | `1` | 0 - 10000 |
| `publishCount` | 每次发布的消息数量 | `1` | 1 - 1000 |
| `publishPeriod` | 发布间隔（毫秒） | `1` | 0 - 60000 |

## 使用场景

### 发布压测（Publish）

测试 MQTT Broker 的消息发布能力：

```yaml
scenario: publish
host: 127.0.0.1
port: 1883
publish:
  connections: 2000    # 2000 个并发发布者
  publishCount: 10     # 每次发布 10 条消息
  period: 0            # 最大吞吐量
```

### 订阅压测（Subscribe）

测试 MQTT Broker 的消息分发能力：

```yaml
scenario: subscribe
host: 127.0.0.1
port: 1883
subscribe:
  connections: 5000    # 5000 个订阅者
  publisherCount: 10   # 10 个发布者
  publishCount: 1
  publishPeriod: 1
```

### 仅测试订阅性能

设置 `publisherCount: 0` 可以仅创建订阅者而不发送消息，用于测试订阅连接建立的速度：

```yaml
subscribe:
  connections: 5000
  publisherCount: 0    # 不启动发布者
```

## 性能调优建议

### 1. 压测客户端调优

- **增加连接数**: 从小值（如 100）开始，逐步增加，找到Broker的处理极限
- **调整 QoS**: QoS 级别越高，Broker 处理开销越大
- **批量发布**: 增大 `publishCount` 可减少网络往返次数

### 2. 服务器端配置

确保 MQTT Broker 满足压测需求：

- 确保服务器硬件资源充足（CPU、内存、网络）
- 调整 JVM 堆内存大小以适应高并发连接
- 确保网络带宽足够（千兆网卡推荐）

### 3. 监控指标

压测时关注以下指标：

| 指标 | 说明 |
|------|------|
| `total` | 5秒内的消息总数 |
| `TPS` | 每秒处理消息数 |
| CPU 使用率 | Broker 所在服务器 CPU 占用 |
| 内存使用 | JVM 堆内存占用 |

## 分布式压测建议

⚡ **重要提示**: 为了准确测试系统极限，建议采用分布式部署架构：

```
+------------------+         +------------------+
|   压测节点        |  网络   |   Broker 节点    |
|  (bench-plugin)  | ======> |   (待测服务)      |
+------------------+         +------------------+
```

**为什么需要分布式部署？**

- 避免资源争抢：压测插件本身会消耗 CPU、内存、网络资源，与 Broker 同节点部署会导致结果不准确
- 减少波动：单节点部署时，压测客户端和 Broker 会竞争同一网卡、CPU 核心，造成 TPS 波动较大
- 真实模拟：分布式部署更接近真实客户端场景

**推荐部署方式：**

1. 在**另一台机器**或**另一个容器**中部署包含 bench-plugin 的 MQTT Broker
2. 将压测节点的 `host` 配置指向目标 Broker 的 IP 地址
3. 两台机器配置相近的资源，以便准确评估单节点处理能力

> 💡 **提示**: 如果只有单台机器，可以通过 Docker 容器部署两个 MQTT Broker 实例进行隔离。

## 注意事项

⚠️ **安全警告**:

- 压测会对服务器产生较大负载，请确保在**测试环境**中进行
- 压测前请告知相关人员，避免影响正常服务
- 建议使用独立测试 Broker，避免生产环境受损

⚠️ **资源限制**:

- 确保服务器可以承受预期的并发连接数
- 客户端数量可能受限于系统文件描述符限制（ulimit -n）
- 高并发压测时建议使用多台压测机器
